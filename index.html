CHAPTER 01

컴퓨터의 언어

모든 컴퓨터는 0과 1만 이해합니다. 전기가 흐르면 1, 흐르지 않으면 0. 이 단순한 원리로 어떻게 문자, 이미지, 영상, 소리를 표현하는지 — 직접 손으로 만져보며 배웁니다.

컴퓨터는 문자를 저장할 수 없습니다. 대신 숫자로 변환해 저장합니다. 'A'는 65, 'a'는 97 — 이 약속이 ASCII입니다. Unicode는 전 세계 문자를 담기 위해 만들어졌습니다.

💡 ASCII는 7비트(0~127)로 128개 문자를 정의합니다. 영문 대소문자, 숫자, 특수문자. 한글은 UTF-8에서 3바이트(24비트)가 필요합니다.

배열 vs 링크드 리스트

배열은 연속된 메모리 공간에 데이터를 저장합니다. 링크드 리스트는 각 노드가 다음 노드의 주소를 가집니다. 삽입/삭제 속도가 완전히 다릅니다.

📦 배열 (Array)
인덱스 접근 O(1) 중간 삽입 O(n) 중간 삭제 O(n)
🔗 링크드 리스트 (Linked List)
인덱스 접근 O(n) 앞에 삽입 O(1) 앞에 삭제 O(1)
💡 배열: 인덱스로 바로 접근 가능(O(1))하지만 중간 삽입 시 뒤 원소를 모두 밀어야 합니다. 링크드 리스트: 포인터만 바꾸면 O(1) 삽입 가능하지만 탐색은 처음부터 순서대로 O(n).

이진 탐색 트리 (Binary Search Tree)

왼쪽 자식은 항상 부모보다 작고, 오른쪽 자식은 항상 큽니다. 이 규칙 덕분에 100만 개 데이터에서도 최대 20번만 비교하면 원하는 값을 찾을 수 있습니다(log₂ 1,000,000 ≈ 20).

숫자를 입력하고 삽입해보세요.

Ch.2 자료구조

해시 테이블 (Hash Table)

파이썬의 dict, 자바스크립트의 Object가 바로 해시 테이블입니다. 키(key)를 해시 함수로 변환해 배열 인덱스를 구하므로 평균 O(1)에 데이터를 저장하고 찾습니다. 충돌(Collision)이 생기면 체이닝으로 해결합니다.

hash("key") = (각 문자 코드 합계) % 8 = 버킷 번호
💡 해시 충돌(서로 다른 키가 같은 버킷에 배정): 체이닝(Chaining) 방식으로 같은 버킷에 연결 리스트로 이어붙입니다. 충돌이 많아지면 탐색이 O(n)으로 느려지므로, 버킷 수와 데이터 수의 비율(Load Factor)을 0.7 이하로 유지합니다.

재귀 (Recursion)

함수가 자기 자신을 호출하는 것. 수학의 귀납법처럼 "작은 문제를 풀 수 있으면 큰 문제도 풀 수 있다"는 아이디어입니다. 콜 스택이 어떻게 쌓이고 풀리는지 확인하세요.

n = 5 → factorial(n) = 120
↓ 콜 스택 (아래=먼저 호출, 위=나중에 호출)
💡 재귀의 핵심: ① 기저 조건(Base Case): factorial(0) = 1 — 재귀를 멈출 조건. ② 재귀 조건(Recursive Case): factorial(n) = n × factorial(n-1). 기저 조건 없이 재귀 호출하면 스택 오버플로우!

운영체제

운영체제(OS)는 하드웨어와 프로그램 사이의 중재자입니다. 메모리를 나눠주고, CPU 시간을 배분하고, 프로그램들이 서로 방해하지 않게 관리합니다. Windows, macOS, Linux가 모두 OS입니다.

메모리 관리가상 메모리프로세스 스케줄링데드락
Ch.4 운영체제

메모리 관리 (Memory Management)

프로그램을 실행하면 OS가 RAM의 빈 공간을 찾아 할당해줍니다. 할당과 해제를 반복하다 보면 작은 빈 공간들이 흩어지는 "단편화(Fragmentation)"가 발생합니다.

메모리 할당과 해제를 반복하면 외부 단편화가 발생합니다. "압축" 버튼으로 빈 공간을 모을 수 있지만, 실행 중인 프로세스를 이동시켜야 해서 비용이 큽니다.

OSI 7계층 모델

네트워크 통신을 7개 계층으로 분리해 표준화한 모델입니다. 각 계층은 바로 아래 계층의 서비스를 사용하고 위 계층에 서비스를 제공합니다. 계층을 클릭해 상세 내용을 확인하세요.

💡 데이터 전송 시 각 계층을 거치며 헤더(Header)가 추가됩니다 — 이를 캡슐화(Encapsulation)라 합니다. 수신 측에서는 반대로 역캡슐화(Decapsulation)하며 헤더를 벗겨냅니다.
Ch.6 네트워크

TCP 3-way 핸드셰이크

HTTP, FTP, SSH 등 신뢰성이 필요한 통신은 TCP 위에서 동작합니다. TCP는 데이터를 보내기 전 먼저 연결을 맺습니다. 그 과정이 바로 3-way Handshake — 3번의 패킷 교환으로 연결을 확립합니다.

💡 왜 3번인가? SYN(연결 요청) + SYN-ACK(수락+요청) + ACK(확인). 2번이면 서버가 클라이언트의 수신 능력을 확인 못 하고, 4번은 불필요합니다. UDP는 핸드셰이크 없이 바로 전송해서 빠르지만 신뢰성이 없습니다(스트리밍, 게임에 사용).

HTTP 요청 / 응답

웹 브라우저와 서버가 대화하는 규칙입니다. 브라우저가 "GET /index.html"이라 요청하면 서버가 "200 OK"와 함께 HTML을 돌려줍니다. 이 단순한 규칙 위에 모든 웹이 만들어져 있습니다.

DNS 조회 과정

도메인 이름(google.com)을 IP 주소(142.250.196.46)로 변환하는 과정입니다. 인터넷의 전화번호부 같은 시스템. 브라우저에 주소를 입력하는 순간 이 과정이 일어납니다.

💡 DNS 캐싱: 한번 조회한 결과는 TTL(Time To Live) 동안 브라우저와 OS에 캐시됩니다. 그래서 두 번째 방문은 훨씬 빠릅니다. DNS 서버를 8.8.8.8(Google), 1.1.1.1(Cloudflare)로 바꾸면 인터넷이 빨라질 수 있습니다.

암호화 — 시저 암호 (Caesar Cipher)

가장 단순한 고전 암호입니다. 각 문자를 알파벳에서 N칸씩 밀어 암호화합니다. 시프트가 3이면 A→D, B→E. 실제 현대 암호(AES, RSA)도 이 아이디어의 수학적 발전입니다.

시프트: 3
PLAINTEXT (평문)
CIPHERTEXT (암호문)
알파벳 매핑표
💡 시저 암호의 약점: 경우의 수가 25가지뿐이라 모든 경우를 시도하면(브루트포스) 즉시 해독됩니다. 현대 AES-256은 2²⁵⁶가지 키가 존재해 우주의 나이보다 긴 시간이 필요합니다.
